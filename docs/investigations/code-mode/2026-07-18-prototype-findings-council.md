# Code Mode Prototype — Findings Council Synthesis (2026-07-18)

- **Findings under review:** `docs/prototypes/findings/2026-07-18-code-mode-client-side-prototype.md`
- **Gate definition:** `docs/investigations/code-mode/2026-07-18-digest-brief.md` (R12 criteria, §3)
- **Artifacts independently re-checked by reviewers:** `.worktrees/code-mode-proto/prototype-code-mode/out/metrics.json`, `run-battery.ts`, retained/raw run artifacts, and original session records.

## 1. Council roster and positions

- **kimi-k3** (confidence: high). Verified every published table number against `metrics.json` and confirmed the mode-664/EACCES bin friction in the recorded `binFriction` block — the arithmetic is honest, the interpretation is shakier. Three problems weaken the headline claims: the managed arm retained 0/0 on all five tasks and was never diagnosed, so the digest's central comparison is unanswered; self-contamination leaked the exclusion filter (2 of the 4 "hand decisive" SDK hits and at least 2 of 8 packaging hits come from omp session `2026-07-18T01-10-36`, the prototype's own build session), and the SDK 4/4 hand score counts this contamination while the adoption arm hand-zeroed the equivalent — an inconsistent standard; pagination was demonstrated only with `maxResults: 10` pages, never against a realistic single page (default 50, ceiling 200). The final "do not productize `session_search_code` now" conclusion is correct and, if anything, stronger than stated once the SDK numbers are decontaminated.
- **codex-56-sol-x-high** (confidence: high). Inspected the full findings and digest, the 88-line harness, metrics, retained hits, raw artifacts, and several original session records. The prototype proves repo-local SDK feasibility and mechanical filtering, but four substantive gates — fanout replacement, evidence-preserving byte savings, pagination value, and precision — remain insufficiently supported. The recommendation not to ship is prudent, but the evidence is inconclusive rather than affirmatively negative on Code Mode product value; the decision should be framed as "do not ship on this evidence; rerun a fair gate."
- **digest-brief-gate-reviewer (fable)** (confidence: medium-high). The findings doc honestly reports mixed evidence: the client-side generated TypeScript prototype convincingly demonstrated programmable fanout, cursor pagination, and large pre-context byte reductions, but it does not clear the digest brief's gate for productizing a `session_search_code` frontend, because the strongest motivating task (native-lane adoption) surfaced no hand-scored organic adoption evidence and the raw-FFF parsing path proved brittle. Registered no disagreements with the findings doc.
- **codex-56-sol-high** (confidence: high). Read the findings and digest fully and inspected `run-battery.ts` plus metrics/raw artifacts. The prototype establishes SDK and orchestration feasibility, but several gate conclusions exceed the evidence. The strongest defects are an underpowered managed arm, incomplete contamination exclusion, post-hoc hand scores absent from metrics, and no pagination experiment showing an answer changed. The do-not-productize recommendation remains defensible, but chiefly because comparative benefit, precision, and answer-changing pagination were not demonstrated — not because the prototype disproved Code Mode.

## 2. Agreements — what the council collectively accepts the prototype proved

All four reviewers converge on the following, all traceable to `metrics.json` / the findings table:

- **Repo-local client-side SDK feasibility.** A TypeScript script using `Client` + `StdioClientTransport` from the already-installed `@modelcontextprotocol/sdk` can spawn `node dist/native-server.js`, discover the seven healthy sources (`codex`, `claude`, `pi`, `cursor`, `hermes`, `gemini`, `omp`), and call the native FFF tools.
- **No infrastructure failure under light load.** 65 generated native calls + 1 `fff_native_capabilities` call across 5 tasks completed with zero budget, timeout, concurrency, or 4 MiB failures (budgets in capabilities: 256 attempted, 4 concurrent, 50 default / 200 max results, 4,194,304 bytes). codex-56-sol-high notes the same native process actually received 101 calls total (35 manual + 65 generated + 1 capabilities), also without failure.
- **Mechanical orchestration is compact.** One shared `run-battery.ts` of 88 lines (87 via `wc -l` per kimi-k3's re-check) handled connect, capabilities, managed comparison, manual native first pages, generated fanout, cursor pagination, filtering, byte measurement, and artifact capture — under the digest's ~150 LOC ceiling.
- **Large serialized-byte reductions are real and reproducible.** SDK 29,221 → 1,086 bytes (96.28%), packaging 32,549 → 2,619 (91.95%), root-wide 26,229 → 623 (97.63%); adoption 25,739 → 961 (96.27%); R13 28,046 → 493 (98.24%). kimi-k3 confirmed metrics.json matches the published table on every cell.
- **Cursor pagination is mechanically drivable from code** and surfaced additional retained lexical hits beyond page one (paginationNew: SDK 2, R13 2, packaging 7, root-wide 1, adoption 2 — per the findings table and metrics.json).
- **The direct-bin failure mode predicted by the 2026-07-17 eval is confirmed in-artifact** for the worktree build: `dist/native-server.js` mode `664`, direct execution `EACCES`, `node dist/native-server.js` works.
- **Raw FFF presentation-text parsing is genuinely brittle** — the strongest negative evidence of the experiment. Automated "precision" was 1.0 everywhere, while post-run hand scoring found only 9 of 19 generated retained hits decisive (adoption 0/3, R13 0/2, SDK 4/4, packaging 3/8, root-wide 2/2). The findings doc deserves credit for reporting hand-score corrections that undercut its own automated metrics.
- **No organic post-ship native-lane adoption evidence was observed** in this run; the 3 retained adoption hits are an eval/planning codex session and the digest's own ProbeAgent session.
- **The do-not-productize-now conclusion stands.** All four reviewers accept the findings doc's bottom line; two (kimi-k3, codex-56-sol-high) would strengthen or reframe its rationale, none would reverse it.

## 3. Disagreements — where reviewers split

- **Fanout replacement.** digest-brief-gate-reviewer scored it a clean "pass" (fanout surfaced decisive hits that would otherwise need serial round-trips). codex-56-sol-x-high scored it "insufficient": 13 sequential calls vs 7 manual first-page calls wraps agent-visible calls but reduces no backend calls, exercises no concurrency, and rests on unsupported hand scoring plus a handicapped managed baseline. kimi-k3 and codex-56-sol-high landed between: mechanically supported, comparative answerability unproven (the manual native arm already answered SDK 2/2, packaging 1/1, root-wide 1/1 from first pages only).
- **Byte savings on 3 tasks.** The findings doc claims "Supported on 3 tasks"; digest-brief-gate-reviewer and codex-56-sol-high accept this narrowly. kimi-k3 disputes it: after decontamination only root-wide (2/2, 97.63%) is clean — SDK's preserved decisive evidence halves to 2/4, and packaging's generated arm was less byte-efficient than manual native on the decisive hit (2,619 vs 354 bytes after filter). codex-56-sol-x-high adds that discarded hits were never recall-audited (102/106 SDK, 114/122 packaging, 90/92 root-wide parsed hits discarded unscored), so evidence preservation is unverified even where hand scores look good.
- **Pagination value.** The findings doc says "Supported, but modest"; digest-brief-gate-reviewer concurs (pass). The other three reviewers all say insufficient: the 2 SDK pagination-only hits are the prototype's own build session (self-contamination), root-wide's 1 added hit was a pre-ship `__advisor.jsonl` line / planning mention when the task was already answerable from page one, packaging's 7 include ~4–5 non-decisive lexical hits, and both native arms used `maxResults: 10` — so the experiment shows pagination beats a 10-result page, not the default 50 or 200 ceiling a manual agent would get in one call. No answer changed on any recorded task.
- **How bad the managed-lane hole is.** kimi-k3: the managed arm retained literally nothing (2-byte `[]` on all 5 tasks) and was never diagnosed — the doc's claim that managed output "was often dominated by current/prototype-session contamination" is asserted without evidence; the correct statement is "the managed comparison failed and was not diagnosed," a hole in the experiment, not a data point. codex-56-sol-x-high and codex-56-sol-high explain the hole: the managed arm got one long natural-language query without explicit literal `queries`, no `callerSession` demotion, no candidate-group continuation, and no `more.evidence` follow-ups — so zero retained managed output does not mean the managed lane lacked answerable evidence. digest-brief-gate-reviewer did not flag this.
- **Strength of the adoption negative.** kimi-k3 and codex-56-sol-high: the ~1-day window (date regex `/2026-07-1[78]/`) for a lane that shipped 2026-07-17, with n=3 retained, is weak evidence — "no evidence observed," re-confirming the digest's prior null, not meaningful negative adoption evidence. The findings doc states the negative slightly more strongly than the design supports. digest-brief-gate-reviewer treated the null as a gate-relevant fail.
- **Mode-664 packaging conclusion.** All reviewers accept the worktree-artifact measurement. codex-56-sol-x-high and codex-56-sol-high caution against generalizing it: a retained source session records installed bins as mode 777, installed dist files as 775, and successful exit 0 — so the finding is local build-artifact friction, not yet a published-install defect.
- **"Raw FFF text is not a stable enough contract."** codex-56-sol-x-high and codex-56-sol-high: the experiment showed semantic and parsing brittleness, not instability across versions or schema changes — the findings doc's wording overstates what was tested.
- **One positive understatement.** codex-56-sol-x-high: the findings understate that repo-local low-level SDK connection and FFF-preserving composition were straightforward enough to justify a *documented experimental client pattern*, even though they do not justify a new shipped frontend.

## 5. What the prototype actually proved — the council's stress-tested restatement

**Survived review:**

1. Client-side generated TypeScript against `agent-session-search-native-mcp` is mechanically viable today, in-repo, with installed dependencies, in ~88 lines of shared harness code. This justifies documenting an experimental client-side recipe; it does not by itself justify a shipped tool.
2. Client-side lexical filtering produces very large serialized-byte reductions against raw native envelopes (91.95%–98.24% across the five generated arms), reproducibly.
3. Raw FFF presentation text is a brittle extraction interface: lexical filtering over it produced plausible false positives at a rate (10/19 retained hits non-decisive by hand scoring) that would be unacceptable for an unattended product surface. This is the experiment's most robust product-relevant finding.
4. The worktree's `dist/native-server.js` is not directly executable (mode 664/EACCES) — confirming the eval's prediction in-artifact, scoped to the local build.
5. Within its light-load envelope (≤101 sequential calls, small results), the native lane's budgets, concurrency limit, and 4 MiB ceiling did not interfere.

**Did not survive review (findings claims narrowed or rejected):**

- *"Byte savings supported on 3 tasks."* Narrowed: cleanly supported on 1 task (root-wide, 2/2, 97.63%). SDK's 4/4 includes 2 self-contaminated hits; packaging's generated arm was less efficient than manual native on the decisive hit (2,619 vs 354 bytes) with only 3/8 decisive.
- *"Pagination value: supported, but modest."* Rejected as insufficient: after decontamination, pagination added at most 1 hand-decisive hit (root-wide) on an already-answered task, and the stacked 10-result pages mean the experiment never tested pagination against a realistic single page.
- *"Precision: mixed."* Narrowed to "not validly measured": the automated metric is circular, the hand scores are unrecorded single-reviewer judgments, and the raw-output baseline was never scored.
- *"Managed output was often dominated by current/prototype-session contamination."* Not evidenced: the managed arm retained 0/0 on all 5 tasks and was never diagnosed; the managed comparison failed and the failure is unexplained.
- *"Organic native-lane adoption still did not appear."* Softened: no adoption evidence observed inside a ~1-day, contamination-prone window with n=3 retained — absence of evidence, re-confirming the digest's prior null, not proof of non-adoption.
- *"Raw FFF presentation text is not a stable enough contract."* Softened: the experiment demonstrated parsing/semantic brittleness at a fixed version, not longitudinal contract instability.

## 6. Methodology issues future prototype rounds must fix

1. **Diagnose or fairly arm the managed lane.** It received one long natural-language query with no explicit literal `queries`, no `callerSession` demotion, no candidate-group continuation, and no `more.evidence`/`more.groupCandidates` follow-ups; its 0/0 × 5 result was never explained. The central comparison of the experiment is unanswered.
2. **Fix contamination exclusion.** Exclusions scanned only path + 500-char previews for a few tokens and missed top-level OMP session files and ProbeAgent records; 6 of 19 generated retained hits (codex-56-sol-high) — including both SDK pagination-only hits — came from contemporaneous pipeline sessions, and decontamination standards were applied inconsistently across arms (adoption hand-zeroed equivalent contamination; SDK scored it 4/4).
3. **Remove circular metrics.** `decisive = retained.length` makes automated precision 1.0 wherever anything is retained. Hand scores need a recorded rubric, per-hit annotations, and ideally independent assessors, stored as artifacts.
4. **Unstack the pagination deck.** Test a single call at the default 50 and ceiling 200 page sizes as arms; add a manual-pagination arm; record whether any answer actually changes. Remove or justify the arbitrary 3-page-per-source cap (recall is currently an unmeasured lower bound, and "13 calls" is partly an artifact of the cap).
5. **Audit recall of discarded hits.** Only retained hits were hand-scored; 88/91 adoption, 107/109 R13, 102/106 SDK, 114/122 packaging, and 90/92 root-wide parsed hits were discarded unscored, so evidence preservation is unverified. Define a ground-truth evidence set per task.
6. **Score the raw baseline.** Raw native output precision was never hand-scored, so "improvement over raw" has no denominator.
7. **Stop baking known answers into the filter.** Each task's decisive list hardcoded the strings the author already expected (`multi_grep_recall_probe_failed`, `mode 664`); this demonstrates retrieval of known answers, not generalizable filtering.
8. **Measure what agents actually pay.** Byte accounting compares full serialized raw envelopes (plus stderr for managed) against a tiny retained-array JSON; it omits generated-code/prompt cost, SDK/protocol overhead, final combined stdout, and tokenization. Report end-to-end context/token cost per arm.
9. **Freeze the corpus and randomize.** The corpus was live and contained the experiment's own sessions, arms ran in fixed order, and there was no snapshot, pre-experiment cutoff, repeated run, or stability hash — exact reruns need not reproduce the reported hits.
10. **Scope packaging claims to what was tested.** The mode-664 check ran against the uninstalled worktree artifact; test a packed/installed artifact before claiming product-level packaging friction (available session evidence shows installed bins 777 / dist 775 / exit 0). Also exercise concurrency, the 4-call limit, near-budget behavior, and timeouts if robustness claims are wanted.
11. **Fix metadata-naive filters.** The adoption date restriction was a regex over path/text, and project scoping required the literal `agent-session-search` in path/text — both create false inclusions/exclusions (e.g. Codex sessions whose dated paths lack the workspace name). Parse session metadata instead.
12. **Count ergonomics honestly.** The 88-LOC figure uses densely packed lines and excludes probe.ts, call-probe.ts, setup/debugging iterations, and the external hand-scoring labor that dominated the conclusions.

## 7. R12 gate evidence assessment

### Evidence FOR productizing Code Mode (`session_search_code`) now — **strength: weak**

- Repo-local client-side SDK composition against the native lane works with installed dependencies: 65 generated calls + 1 capabilities call, zero budget/timeout/concurrency/4 MiB failures (capabilities budgets: 256 attempted, 4 concurrent, 4,194,304 bytes).
- One 88-line shared harness covers capabilities discovery, 7-source fanout, cursor pagination, filtering, managed comparison, and metrics — under the ~150 LOC ergonomics ceiling, the one gate criterion that passes cleanly.
- Serialized-byte reductions of 91.95%–98.24% vs raw native envelopes are real and reproducible on all five generated arms.
- Pagination and filtering are mechanically executable from generated code (paginationNew hits recorded on all five tasks).
- This supports a *documented advanced-agent client pattern* — but no reviewer found evidence that generated code beats the managed lane an agent already has, because that comparison was never validly run. Feasibility evidence is not gate evidence.

### Evidence AGAINST productizing now — **strength: moderate**

- **Comparative benefit unproven:** the managed arm failed (0/0 × 5, 2-byte `[]`) undiagnosed and handicapped; the digest's central question — does code beat the managed lane? — is unanswered.
- **Precision unproven and partly contradicted:** automated precision is circular; hand-decisive totals are 9/19, with 0/3 (adoption) and 0/2 (R13) on the two most product-relevant tasks; raw-baseline precision never measured.
- **Self-contamination invalidated the strongest positive cells:** 2 of 4 SDK "decisive" hits and ≥2 of 8 packaging hits are the prototype's own build session (omp `2026-07-18T01-10-36`); 6/19 retained hits from contemporaneous pipeline sessions overall.
- **Pagination value not demonstrated:** no answer changed; demonstration used stacked 10-result pages; decontaminated incremental value ≈ 1 hand-decisive hit on an already-answered task.
- **Raw presentation-text parsing is brittle** — the best-evidenced substantive negative: lexical filtering over it yields plausible false positives requiring human review, so an unattended `session_search_code` over the current native output contract would ship exactly this brittleness.
- **Direct-bin friction is real** in the worktree artifact (mode 664, EACCES).
- Two caveats keep this at *moderate* rather than *strong*: the adoption null is a ~1-day, n=3, contamination-prone window (absence of evidence, not evidence of absence), and the bin friction is a local build-artifact result that available session evidence (installed bins 777, dist 775, exit 0) suggests may not reproduce from a packed install. The evidence supports "do not ship on this evidence; rerun a fair gate" — it does not disprove Code Mode's intrinsic value.

## 8. Implications for plan drafting

The plan council must address:

1. **Do not open a `session_search_code` implementation track on this evidence.** Frame the R12 gate as *not yet evaluated fairly*: the managed comparison must be rerun with explicit literal `queries`, `callerSession` demotion, and candidate/evidence follow-ups before any code-vs-managed verdict is meaningful.
2. **Prioritize a typed/structured native result contract over any Code Mode frontend.** The single most robust finding is that raw FFF presentation text is too brittle to filter programmatically (10/19 false positives). A typed result mode would help the managed lane, the native lane, and any future client-side pattern simultaneously — and the findings doc's own promotion recommendation points the same way (small documented native-lane call helper or typed CLI/JSON result mode, not this harness).
3. **Consider the "documented client-side pattern" option explicitly.** The strongest positive result is that a compact SDK recipe works today at zero product cost. A short doc/recipe (plus fixing the `dist` executable bit at build/pack time) may capture most of the real value without a new shipped surface. This is a genuinely different option from both "ship a tool" and "defer."
4. **Treat the adoption gate as still-open, not failed.** Any future gate run needs a longer, contamination-controlled observation window (frozen corpus or pre-experiment cutoff, metadata-based date/project scoping, exclusion of pipeline sessions by session-id, not preview tokens).
5. **Define token economics before the next gate.** Byte-vs-raw-envelope reductions are a strawman; the plan needs an end-to-end context/token cost model per arm (generated code + protocol overhead + synthesis included) with explicit pass thresholds, per digest §5's open question.
6. **Fix the packaging question at the right layer.** Reproduce mode-664/EACCES from a packed/installed artifact before treating it as a release defect; if confirmed, the fix belongs in build/packaging, not in a Code Mode scope.
7. **Mandate methodology upgrades for any rerun:** non-circular precision instrumentation with recorded rubrics, recall audits of discarded hits, raw-baseline scoring, realistic page-size arms, ground-truth evidence sets, and frozen-corpus reproducibility (§6).
8. **Keep server-side Code Mode out of scope.** Nothing in this evidence challenges the digest's sequencing (client-side first, server-side sandboxed execution out of scope, arbitrary code execution remains a `DESIGN.md` non-goal).
9. **If a helper is promoted, promote a recipe, not the harness.** The 88-line battery is amortized, known-answer, task-specific infrastructure; per the prototype lifecycle rules it should not be merged as a documented tool.
